## Why position matters

Self-attention, as we saw in Post 2, is a **set operation**. Take the query / key / value vectors for "the cat sat on the mat" and shuffle the words — the attention matrix shuffles with them, but no information is destroyed. The transformer block treats its input as a bag of vectors; nothing in the math knows that "the" came before "cat".

For most language tasks that's catastrophic. So we have to inject position information somewhere — and the *how* turns out to matter a lot. It's the difference between a model that handles 4K tokens and one that handles 1M.

This post is about three ideas, in order: the original **sinusoidal** scheme from Attention Is All You Need, the **RoPE** rotation trick that powers nearly every modern LLM, and **YaRN**, the cheap fine-tune that stretches RoPE-trained models to 128K+ context windows.

## The intuition: positions as bit patterns

Before any equations, here's the analogy that makes everything else click.

How do you uniquely identify positions 0, 1, 2, ..., N? In binary: each position gets a bit pattern. The lowest-order bit alternates every step (0, 1, 0, 1, ...), the next bit every two steps (0, 0, 1, 1, ...), the next every four, and so on. **Each position is a unique combination of fast-changing and slow-changing bits.**

Sinusoidal positional encoding is the same idea, but with continuous waves instead of bits. Instead of bits flipping at frequencies 1, 2, 4, 8, you have sines and cosines oscillating at a smooth spectrum of frequencies — fast ones for "where am I locally" and slow ones for "where am I in the sequence overall".

Now the equation:

The `10000^(2i/d)` denominator spreads the frequencies geometrically: dimension 0 oscillates fastest, dimension `d-1` oscillates slowest. Visualised, the encoding for a position is a column-slice of this heatmap:

They're then concatenated to form each of the positional encoding vectors. </Text>

The encoding is **added to the token embedding** before the first attention layer. Every attention head from there on sees `embedding + position` — and because each position has a unique sin/cos signature, the heads can learn to behave order-aware.

## The two limitations

Sinusoidal works, but it has two flaws that show up at scale.

**It encodes absolute position.** A model trained this way has to *learn* relative behaviour ("token at position 17 attending to token at position 14") implicitly from absolute encodings. That's solvable but indirect — and it bakes the assumption that positions 0, 1, 2, ... mean fixed things, which is brittle.

**It doesn't extrapolate.** If you train the model on sequences up to 4K tokens, then try to run it on a 32K-token input, positions 4K–32K hit sin/cos values the model has literally never seen. Behaviour falls off a cliff: garbled output, attention collapsing to the start of the sequence, perplexity exploding.

The fix for the first problem is RoPE. The fix for the second is YaRN.

## RoPE: rotate, don't add

Rotary Position Embedding (Su et al., 2021) replaces "add a position vector to the embedding" with "rotate the embedding by an angle proportional to position". Crucially, the rotation is applied to the **Q and K vectors inside attention**, not to the embedding before it.

Here's the move. Treat each pair of dimensions in Q (and K) as a 2D point. For a token at position `pos`, rotate that 2D point by angle `θ_{pos,i} = pos / 10000^(2i/d)`:

  <Media src="/images/blog/llm-internals-06-rope-yarn-context-window-extension/eq-rope-rotation.svg" alt="2D rotation matrix R(θ) with cos and sin entries; θ depends on position and dimension index" style={{ maxWidth: "440px", width: "100%" }} />
</Column>
<Text variant="label-default-xs" onBackground="neutral-weak" align="center">
  A rotation matrix applied to one 2D pair of (Q, K) dimensions. Different pairs get different frequencies — same geometric spread as sinusoidal.
</Text>

Now the magic. When attention computes the dot product between a query at position `n` and a key at position `m`:

<Column horizontal="center" fillWidth>
  <Media src="/images/blog/llm-internals-06-rope-yarn-context-window-extension/eq-rope-relative.svg" alt="Dot product of two rotated vectors equals q transposed times R of (theta_m minus theta_n) times k, which is a function only of m minus n" style={{ maxWidth: "560px", width: "100%" }} />
</Column>
<Text variant="label-default-xs" onBackground="neutral-weak" align="center">
  Two rotations multiplied are a single rotation by their angle difference. So Q·K only sees the *relative* offset between positions.
</Text>

That's it. The dot product depends only on `m - n` — the **relative distance** between the two tokens — never on their absolute positions. We didn't have to learn this; the geometry forces it.

<Text variant="label-default-xs" onBackground="neutral-weak" align="center">
  Source: <SmartLink href="https://arxiv.org/abs/2104.09864">Su et al. 2021, "RoFormer"</SmartLink>, Figure 1. ## A worked numerical example

Pick a 2D pair of dimensions (i.e. dimension index `i = 0`, so `θ = pos / 10000^0 = pos` radians). For a query vector `q = (1, 0)`:

<Column horizontal="center" fillWidth>
  <Media src="/images/blog/llm-internals-06-rope-yarn-context-window-extension/eq-rope-worked.svg" alt="At position 0, theta is 0 and the rotation is identity. At position 100, theta is 100 radians and the rotation matrix is cos 100 negative sin 100 over sin 100 cos 100, applied to q." style={{ maxWidth: "620px", width: "100%" }} />
</Column>
<Text variant="label-default-xs" onBackground="neutral-weak" align="center">
  Numerically: cos 100 ≈ 0.862, sin 100 ≈ -0.506 (radians, modulo 2π). So at position 100, the same q-vector points in a meaningfully different direction than it did at position 0.
</Text>

Now consider two queries at positions `n = 0` and `n = 100`, and the *same* key vector `k = (1, 0)` placed at positions `m = 1` and `m = 101`. The angle gap is identical (`m - n = 1` in both cases), so the dot product is identical. The model sees "this key is one position to my right" — regardless of whether it's at position 1 or position 101.

That's the property no amount of additive positional encoding can give you for free.

For higher-frequency dimensions (`i` larger), `θ_pos = pos / 10000^(2i/d)` shrinks fast. At `d = 64`, dimension `i = 31` rotates by `pos / 10000` radians — barely a tick across the entire sequence. So **fast-rotating dims encode local position, slow-rotating dims encode global position** — same bit-frequency intuition as sinusoidal, just expressed in rotation angles.

## The remaining problem: extrapolation still fails

RoPE solved relative-position encoding elegantly. But it did not solve extrapolation.

If a model is trained on sequences up to 4K tokens, the slow-frequency dimensions of RoPE only ever see angles up to `4096 / 10000^(2i/d)` — a small fraction of a full rotation for the slowest dimensions. Push the model to position 32K and those dimensions start landing in angle ranges they never visited during training. Q·K dot products go off-distribution, attention scores misbehave, and quality drops fast.

The naïve fix — just train on longer sequences — works but is expensive. Pre-training a 70B model on 128K-token sequences requires far more memory and compute than 4K. The whole point of context extension is to **make a model trained at 4K usable at 128K with as little extra training as possible**.

## YaRN: not all frequencies are equal

YaRN (Peng et al., 2023) starts from one observation: the trouble is concentrated in the slow-rotating dimensions, because those are the ones that don't complete enough rotations during training. The fast-rotating dimensions wrap around many times even at short context — they've effectively been trained on every angle in [0, 2π).

So treat them differently. YaRN defines a **per-frequency ramp**: scale down the rotation angles a lot for slow dimensions (so they reach reasonable angles even at long positions), leave the fast dimensions alone (they didn't need fixing), and smoothly interpolate in between:

</Column>
<Text variant="label-default-xs" onBackground="neutral-weak" align="center">
  s is the target context-extension factor (e.g. 32× to go from 4K to 128K). Only the slow dimensions are stretched; the fast dimensions are untouched. The middle band is a smooth ramp.
</Text>

<Media src="/images/blog/llm-internals-06-rope-yarn-context-window-extension/yarn-method-summary.png" alt="Diagram comparing positional encoding extension methods: original RoPE, Position Interpolation (uniform stretch), NTK-aware (frequency-dependent), and YaRN (ramp + temperature)" />
<Text variant="label-default-xs" onBackground="neutral-weak" align="center">
  Source: <SmartLink href="https://arxiv.org/abs/2309.00071">Peng et al. 2023, "YaRN"</SmartLink>, Figure 1. Earlier methods (PI, NTK-aware) either stretched all frequencies uniformly or used a single smooth scaling. YaRN's piecewise ramp is what preserves local fidelity.
</Text>

This costs a short fine-tune (typically a few hundred million tokens at the new context length — much cheaper than training from scratch) and it works strikingly well: a 4K-context model becomes a 128K-context model with no quality loss on the original 4K tasks and strong long-context retrieval on the new range.

## Where RoPE and YaRN show up in practice

By 2026, RoPE is the default. Almost no new architecture uses sinusoidal encoding. YaRN (or close variants) is the go-to context-extension recipe whenever a base model needs a longer window than it was originally trained on.

| Model | Positional scheme | Trained context | Extended context |
|---|---|---|---|
| Llama 3 70B | RoPE (base 500K) | 8K | 128K (rope-base scaling) |
| Llama 3.1 / 3.3 | RoPE + custom scaling | 8K | 128K |
| Mistral / Mixtral | RoPE (base 1M) | 32K | — |
| Code Llama | RoPE (base 1M) | 16K | 100K |
| DeepSeek-V3 | RoPE + YaRN | 4K | 128K |
| Qwen 2.5 / 3 | RoPE + YaRN-style scaling | 32K | 1M (Qwen 2.5-1M) |
| GPT-4o, Gemini, Claude | proprietary, presumed RoPE-family | — | 128K – 2M |

Two patterns to notice. First, **rope-base** (the `10000` in the denominator) is increasingly cranked up — Code Llama uses 1,000,000 instead of 10,000, which slows down all frequencies and makes the encoding inherently more long-context friendly. Second, the public open models (Llama, DeepSeek, Qwen) all use some YaRN variant for their longest-context releases. Frontier closed models almost certainly do too — RoPE-family encodings dominate because nothing else has matched their long-context behaviour at training-cost parity.
